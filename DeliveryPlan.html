CIB Unified Ingress: Deliverables & Estimates

Executive Summary

CIB Unified Ingress delivers a single, secure, and observable internet ingress platform across all CIB Lines of Business. Today each LOB manages ingress independently, with inconsistent security controls, duplicated platforms, and no shared visibility. This initiative standardizes that across Markets, Payments, Security Services, and Global Banking. The platform is designed for regional resilience: in the event of a failure anywhere in the request path within a region, traffic automatically fails over to a healthy region, with session continuity maintained through the active/active Session Manager.

Risk Reduction
  • Standardized auth, routing, and policy enforcement across all LOBs
  • Eliminates inconsistent ingress patterns that expand the attack surface
  • Phishing-resistant authentication. Stolen tokens are cryptographically unusable without the client's private key
  • Session revocation propagated globally in under one second; access decisions reflect live permission state without relying on token claims
Cost & Efficiency
  • Decommissions duplicate ingress platforms across LOBs
  • Eliminates per-team engineering overhead maintaining custom solutions
  • Single control plane: one place to define, validate, and deploy routes across all regions
  • GitOps-driven config with policy CI blocks non-compliant routes before they reach production
Resilience & Observability
  • Improved reliability through consistent, well-tested ingress patterns
  • Centralized metrics, logging, and traffic visibility across all LOBs
  • End-to-end distributed tracing from CDN edge to L5 service
  • Multi-region active/active: no single region or cloud is a single point of failure
Strategic Alignment
  • Aligns to firmwide platform strategy for shared infrastructure
  • Enables future capabilities: zero trust, policy-as-code, and automation
  • Extensible policy engine (OPA + SpiceDB) supports evolving entitlement requirements
  • Positions CIB as the reference implementation for ingress across the firm

Adoption: Lines of Business

Initial adopting entities at platform launch. JPMM, JPMDB, JPMA, PDP, and HRL all require the full authentication and authorization pipeline. Gateway type is determined by traffic profile. Envoy for web/browser flows requiring HTTP/2, streaming, and DPoP enforcement; Kong for programmatic API clients requiring consumer identity, rate limiting, and managed API lifecycle. Approved patterns from elsewhere in CIB will be incorporated during P5 route configuration. DPoP token binding Implement DPoP proof generation (browser via WebCrypto API) and verification (gateway). Validate htm, htu, iat, jti claims. Enforce JWK thumbprint binding between proof and access token cnf claim. Replay store with TTL eviction. 6w OAuth flows
Session Manager
Session Manager core Session JWT issuance separate from AuthE access tokens. Sentry routes the auth flow but the Session Manager issues and owns the session JWT independently of both Sentry and AuthE. The session JWT carries identity attributes only: sub, sid, entity, client type (workforce or client), and DPoP thumbprint. Entitlement claims are deliberately excluded. Authorization decisions are resolved dynamically at request time from the ABAC attribute source, ensuring revocations and permission changes take effect immediately without waiting for token expiry. EC key pair generation, JWKS publication, session lifecycle management, and key rotation independent of the IdP. 5w DPoP, OAuth flows
Gateway Auth

Deliverable Description Est. Dependencies & Notes
OPA
OPA coarse policy at L4 (gateway sidecar) Deploy OPA as Policy Decision Point (PDP) sidecar per gateway Pod, called by Envoy via the ext_authz filter which acts as the Policy Enforcement Point (PEP). Implement coarse ABAC Rego policies: route access based on entity, client type, and dynamically resolved entitlement attributes fetched from the CIB ABAC attribute source (Policy Information Point). Attribute lookups are cached per session with a configurable TTL to bound latency while ensuring revocations propagate within the cache window. Policy bundle provides the Rego rules; live attributes come from the PIP at request time. 5w Sidecar pattern, gateway auth layer, ABAC attribute source (PIP) integration. PIP endpoint and attribute schema must be agreed with the owning team before OPA policy authoring begins.
OPA policy bundle authoring and deployment Define bundle structure, versioning, and signing. Rego policy rules encode routing logic and ABAC evaluation patterns — they define which attributes to fetch from the PIP and how to evaluate them, not static entitlement lists. Implement bundle build and publish pipeline. Kafka-notified bundle refresh on push. Drift monitoring alert if L4/L5 instances diverge > 60 seconds. 3w OPA L4, Kafka policy-bundle-push topic, ABAC attribute schema from PIP team
OPA fine-grained policy at L5 via AuthZen API Deploy OPA L5 service exposing IETF AuthZen API. L5 services call AuthZen for resource-level entitlement decisions, receiving allow/deny with structured obligations. OPA evaluates against both the ABAC attribute source (for attribute-based decisions) and SpiceDB (for relationship-based decisions) depending on the service schema. Decoupled from OPA: the policy engine is substitutable without service code changes. 4w Context Propagator, L5 service integration, SpiceDB for services requiring ReBAC
SpiceDB
SpiceDB — L5 ReBAC pattern Deploy SpiceDB as a supported L5 authorization pattern for service teams requiring relationship-based access control (e.g. trading desk membership, client-to-trader relationships, portfolio ownership). The Policy Squad defines reference schema patterns, AuthZen integration guidance, and onboarding documentation. Service team adoption is opt-in based on their specific authorization requirements. SpiceDB is not in the L4 gateway request path. 4w SpiceDB (Authzed) requires technology approval if not already on the CIB approved list — initiate TAC submission early. Context Propagator
Context Propagator Implement trust boundary crossing between internet-origin and internal zones. Strip all client-supplied headers unconditionally. Forward an explicit allowlist of verified identity attributes from the session JWT (sub, sid, entity, client type) as x-auth-* headers to backend services. Entitlement decisions are not forwarded — L5 services make their own authorization calls against the AuthZen API using the forwarded identity. Unknown headers dropped. Implemented as Envoy filter and Kong plugin. 2w OPA L4, gateway auth layer
